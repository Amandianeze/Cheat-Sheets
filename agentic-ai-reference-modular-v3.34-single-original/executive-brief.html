Production reference and market map for agentic AI systems

This document is a practical field guide for designing, choosing, operating, and governing agentic AI systems. Use it to move from product intent to architecture choices, from architecture choices to implementation stacks, and from implementation stacks to production controls.

Who should read what
ExecutivesRead this summary, the brief version, production rules, and risk checklist.
Product leadsStart with product patterns, use-case decisions, and anti-patterns.
EngineersUse the stack map, revised architecture, default stacks, state, tools, and deployment sections.
ML / DS teamsFocus on retrieval, evals, data quality, model gateways, and the ML/DS bridge.
Platform / securityPrioritize observability, identity, policy gates, audit trails, rollback, and freshness controls.
Top 10 production rules
  1. Do not add agents where deterministic workflows are enough.
  2. Separate planning, execution, memory, retrieval, and evaluation.
  3. Treat tools as privileged interfaces with permissions and audit logs.
  4. Log decisions, tool calls, inputs, outputs, and failure paths.
  5. Version prompts, schemas, tools, embeddings, evals, and policies.
  6. Keep human approval on irreversible or high-risk actions.
  7. Make retrieval freshness and source provenance first-class controls.
  8. Design rollback, timeout, and graceful-degradation paths before launch.
  9. Measure task success and operational risk, not only model quality.
  10. Prefer small, inspectable loops over broad opaque autonomy.

What to use, what to avoid, and what changed

In this sectionUse by defaultAvoid by defaultChanged in v3.34

This reference is split into two mental modes: Production Reference for durable system design, and Market Map for fast-changing tools, vendors, protocols, and model-provider choices. Start with the default architecture path, then consult the market map only when a concrete layer needs a product decision.

Durable referenceVolatile market mapProduction-first

Use by default

  • RAG or workflow automation before autonomous agents.
  • A graph/runtime layer when tasks require state, branching, retries, or long-running execution.
  • Hybrid retrieval, explicit state, observability, evals, security boundaries, and launch gates.
  • A model gateway when routing, cost control, fallback, logging, or provider abstraction matters.

Avoid by default

  • Agents without traces, evals, cost budgets, rollback paths, or human approval for high-impact actions.
  • Long-context prompting as a substitute for retrieval, permissions, and provenance.
  • Vendor lock-in before requirements stabilize.
  • Tool catalogs that do not separate default, strong, niche, watch, and deprecated-risk options.

Glossary · operational definitions

How this document uses key terms

These definitions are intentionally operational and alphabetized for lookup. They clarify terms that are often used inconsistently across agent, AI app, infrastructure, graph, retrieval, and ML tooling discussions.

AgentA system that can choose actions over multiple steps using tools, state, memory, feedback, policies, and termination criteria.
Apache AGEA PostgreSQL extension that adds graph database capabilities and Cypher-style graph querying inside a Postgres-centered stack.
CypherA declarative graph query language used heavily with Neo4j to match nodes, relationships, paths, and graph patterns.
DatabricksEnterprise lakehouse and AI platform used as a governed data, ML lifecycle, vector search, and evaluation backbone for agents; complements agent frameworks rather than replacing them.
Entity ResolutionThe process of identifying when different names, aliases, records, or mentions refer to the same real-world entity.
EvaluationMeasurement of quality, reliability, safety, groundedness, cost, latency, and task success before and after release.
GatewayA control layer between apps and model/tool providers for routing, policy, logging, fallback, cost control, and abstraction.
Graph DatabaseA database optimized for storing and querying entities and their relationships, especially when paths, neighborhoods, lineage, or multi-hop connections matter.
Graph TraversalThe act of moving across relationships in a graph to find connected entities, paths, dependencies, permissions, or evidence chains.
GraphRAGA retrieval-augmented generation pattern that retrieves graph entities, relationships, paths, neighborhoods, communities, and provenance instead of only text chunks.
HarnessThe developer-facing wrapper around prompts, tools, tests, state, policies, and runtime behavior.
Knowledge GraphA structured graph of entities, relationships, facts, claims, metadata, and provenance used to support reasoning, retrieval, lineage, and explanation.
MemoryInformation preserved across turns or sessions for personalization, continuity, preferences, learned user context, project history, or long-running task context.
Multi-hop RetrievalRetrieval that requires more than one relationship or reasoning step, such as finding how two entities connect through intermediate documents, systems, people, or events.
Neo4jA graph-native property graph database commonly used for knowledge graphs, GraphRAG, recommendations, entity relationship traversal, fraud/compliance graphs, and dependency analysis.
Node / Relationship / PropertyCore property-graph concepts: nodes represent entities, relationships represent directed connections, and properties store attributes on either nodes or relationships.
OrchestrationThe logic that decides what happens next: routing, branching, retries, delegation, tool use, approvals, and stopping.
Property GraphA graph model where entities are represented as nodes, connections as relationships, and both can carry key-value properties.
ProtocolA shared interface standard that lets models, tools, agents, apps, or UIs interoperate.
ProvenanceTraceable evidence about where data, claims, relationships, retrieved context, tool results, or generated outputs came from.
RuntimeThe execution environment that manages steps, tool calls, retries, persistence, streaming, scheduling, and checkpointing.
SnowflakeEnterprise data cloud and AI platform used for governed SQL analytics, Cortex-powered text-to-SQL/search, internal data apps, and business-facing agent data access; complements orchestration frameworks rather than replacing them.
StateThe explicit machine-readable representation of workflow progress, task variables, checkpoints, and execution context.
Vector IndexA searchable index over embedding vectors used to find semantically similar chunks, entities, nodes, documents, or relationships.

// Anti-Patterns

When not to add agentic complexity

Use this section as a brake. Most production failures come from adding autonomy before the system has observability, evaluation, access control, rollback, and clear user value.

Do not use agents when a workflow is deterministic.Prefer: rules, queues, cron jobs, SQL, or standard workflow orchestration when the task does not require reasoning or ambiguity handling.
Do not add memory before you know what should persist.Prefer: explicit user profiles, scoped preferences, and auditable state before implicit long-term memory.
Do not add multi-agent orchestration to fix unclear requirements.Prefer: one reliable agent with a narrow objective, clear tool contracts, and measurable success criteria.
Do not use RAG as a substitute for content governance.Prefer: curated corpora, ACL-aware indexing, deduplication, freshness metadata, and citation evaluation.
Do not give write tools to unmeasured systems.Prefer: read-only tools, sandbox writes, dry runs, approvals, and rollback plans until evals and monitoring pass.
Do not optimize model choice before measuring task quality.Prefer: a small golden dataset, task-specific evals, latency/cost traces, and side-by-side model regression tests.

For ML & Data Science Practitioners

How your existing skills map to the agentic AI stack — and what's genuinely new

If you know machine learning or data science, you already understand more of this stack than you think. Agentic AI is not a replacement for ML — it's inference-time compute applied to goals rather than tokens. This section maps concepts you know to concepts in this document, so you can navigate the rest of it with confidence.

// Concept mapping: what you know → what this document calls it
Memory
Context window ≈ Working memory / RAM
An LLM's context window is finite (~200k tokens) — everything the model can "see" right now. When it fills, you need external memory (Mem0) or retrieval (RAG). This is why vector stores exist: they extend context beyond what fits in the window.
Retrieval
RAG ≈ Retrieval-augmented inference
RAG (Retrieval-Augmented Generation) is retrieval-augmented inference: instead of embedding all knowledge in model weights, you retrieve relevant documents at inference time and inject them into context. You already know embedding models and nearest-neighbor search — those are the DS-familiar pieces of the RAG stack.
Control loops
ReAct / MCTS / Beam Search ≈ Inference-time search
Control loops are search algorithms run at inference time over goals rather than tokens. ReAct is a greedy search loop. Tree-of-Thought branches and prunes. MCTS (Monte Carlo Tree Search) — which you may know from AlphaGo — is the same algorithm applied to reasoning traces rather than game boards.
Evaluation
Agent eval ≈ Online experiment design
Agent evaluation is closer to online A/B testing of decision policies than offline model benchmarking. Instead of comparing predictions vs. ground truth, you compare trajectories — did the agent take the right sequence of actions? Golden datasets here are (input, expected trajectory) pairs rather than (input, label) pairs.
Observability
MLflow / experiment tracking ≈ Langfuse / LangSmith traces
You know MLflow for experiment tracking. Langfuse and LangSmith are its agentic equivalent — they trace not just model calls but tool invocations, context assembly, retrieval results, and token costs across every step of a run. Same concept, more dimensions.
Features vs. context
Feature store ≈ Context + retrieval layer
In traditional ML, a feature store serves engineered features to a model at serving time. In agentic AI, the retrieval layer plus in-context injection serves the same function — assembling the right facts into context before inference. The "features" are now retrieved passages, structured records, and tool outputs rather than numerical vectors.

The key insight: In traditional ML, your job is to build a model that encodes knowledge at training time and produces predictions at serving time. In agentic AI, your job is to build a system that assembles knowledge at inference time, decides what actions to take, and improves from feedback over time. Your embedding models, retrieval skills, and evaluation instincts are load-bearing in the agentic stack — they're not obsolete, they're promoted.
