Section TL;DR
  • The stack has 16 layers; every layer exists to feed, constrain, observe, or govern the model layer.
  • A chatbot is an agent with fewer layers active - same stack, lower autonomy.
  • Use the complexity spectrum to pick which layers you need before adding tooling.

The production agent stack: layers, inner loop, and cross-cutting planes

Use this as the canonical mental model for placing agent components. The vertical stack describes implementation boundaries; the inner loop describes the agent's behavior; the side planes describe controls that instrument, evaluate, secure, and govern more than one layer at a time.

Layer placement cheat sheet

  • Check the freshness badge or review date for market/vendor claims before making a production decision.

Cross-cutting planes Observability, evaluation, security, governance, privacy, cost, policy, and incident response do not live in one layer. They instrument or constrain the harness, framework, runtime, tools, sandbox, and product workflow.
Inner behavioral loop
Think → Act → Observe
ReAct, plan-and-execute, reflection, evaluator-optimizer, and tree search run inside the framework/runtime. They call outward through tool interfaces and receive observations from the execution substrate.
Boundary rule LiteLLM-style gateways standardize and govern model access; MCP standardizes tool/resource access. MCP exposes a capability; the sandbox, database, browser, filesystem, or SaaS service performs the side effect. LangSmith observes traces across layers; it is not itself the runtime.
Model path vs. tool path
Model path
Agent Framework / Runtime → LiteLLM or AI Gateway → OpenAI, Anthropic, Gemini, Bedrock, Azure OpenAI, Groq, Together, vLLM, Ollama, or another inference backend.
Tool path
Agent Framework / Runtime → MCP or tool interface → sandbox, database, browser, filesystem, SaaS API, queue, or storage system.
Read the stack downward, then read controls sideways: product/control plane decides what may run; the harness starts the agent session; the framework provides reusable agent abstractions; the runtime manages state over time; the capability interface exposes tools/resources/prompts; the execution substrate safely performs side effects. Observability, evals, security, and governance wrap the whole path.

🧠MEMORY LAYERWhat the agent remembers across sessions
Mem0
Mem0 AI · embeddable middleware
Lightweight · extracts + stores memories via API · drops into any existing harness · low overhead · self-hostable · best for adding memory to existing pipelines
// ML/DS insight · continual learning & catastrophic forgettingCross-session memory is continual learning without weight updates — and faces the same core problem: storing new information without overwriting or contradicting prior facts. This is the catastrophic forgetting problem from continual learning research. When a user's preference changes ("I used to prefer formal tone, now I want casual"), naive memory systems either ignore the update (forgetting failure) or overwrite the prior state (interference failure). Mem0 addresses this at the extracted-fact level — deduplicate, reconcile contradictions, merge summaries — rather than updating model weights. This sidesteps gradient-based forgetting but introduces a new problem: contradiction resolution is still hard. Design memory schemas with explicit TTLs for volatile preferences vs. stable facts, and build correction mechanisms so users can delete or amend stored memories.

Data Substrate for Agentic Performance

Agentic AI performance is not only a model problem. The data substrate controls how quickly an agent can assemble context, how accurately it can ground decisions, how safely it can respect permissions, and how well it can learn from traces. Treat relational, vector, graph, time-series, document, and in-memory databases as complementary performance infrastructure rather than interchangeable storage options.

🧬Agent Data Loophow data improves behavior
1 · Request
The user request enters the planner with identity, tenant, session, and policy context. This is where relational permissions and session state prevent unsafe or irrelevant work from starting.
2 · Context assembly
The agent gathers structured records, semantic passages, relationship paths, recent events, artifacts, and cache hits before the model reasons. Better context assembly usually beats larger prompts.
3 · Tool execution
The runtime calls tools with bounded schemas, transaction checks, and fresh data. Relational and in-memory systems keep actions correct and fast; graph and vector systems keep actions grounded.
4 · Response
The answer is produced with citations, provenance, policy filtering, and evidence. The database layer gives the model facts, not just memories.
5 · Feedback
Traces, eval outcomes, user corrections, events, and cost/latency metrics flow back into observability, memory, and future routing decisions.

Graph Schema Design for Agents The highest-leverage decision in a graph system is what to model before you ingest anything. A schema designed for completeness rather than retrieval produces a graph that is expensive to traverse and hard to query.
Model entities as nodes People, organizations, documents, products, systems, concepts, risks, obligations, and events. Each node type gets a label. Node properties store attributes — name, ID, timestamp, confidence, permission tags.
Model facts as relationships WORKS_AT, DEPENDS_ON, OWNED_BY, CITED_IN, APPLIES_TO, CONNECTED_TO. Relationships should be typed, directed, and carry properties: confidence, provenance chunk ID, extraction timestamp. Avoid relationships named RELATED_TO — too vague for traversal.
Keep long text out of nodes Long document text belongs in the vector layer, not on graph nodes. Node properties should be short, structured, and queryable (name, ID, category, date, permission label). The graph and vector layers are complementary — don't duplicate content between them.
Schema design rule: Start from the questions the agent needs to answer, not from the documents you have. Work backwards from "How is A connected to B?" to the relationship type that path requires. Every node type and relationship type in your schema should be traceable to a real retrieval use case.

🐘PostgreSQL Extension Stack for Agentic Systemsretrieval · telemetry · graph · scale · operations

Postgres is not only the relational layer. Through extensions it can become the semantic retrieval layer, lexical search layer, graph layer, time-series layer, scheduler, query-observability layer, distributed SQL layer, in-database ML runtime, and analytical execution surface. Every extension registers its capabilities via PostgreSQL's CREATE EXTENSION mechanism, which runs an install script that may register custom index access methods, background workers, executor hooks, and shared memory segments. Treat extensions as composable capabilities, not random plugins.

pgvector — semantic retrieval
IVFFlat · HNSW · cosine / L2 / dot operators
pgvector adds a native vector column type and distance operators (<->, <#>, <=>) to Postgres. It registers two custom index access methods via IndexAmRoutine: IVFFlat, which partitions vectors into inverted file lists for fast approximate search; and HNSW, which builds a multi-layer navigable small-world graph where each node connects to nearby vectors at multiple zoom levels — enabling fast greedy traversal at query time with high recall. HNSW is generally preferred for production RAG. Combine with standard SQL filters, joins, and transactions without any data sync overhead.
HNSW preferred for RAGSQL + vectors in one DBzero extra infra
pgvectorscale — large-scale ANN
DiskANN · streaming graph · RAM-efficient
pgvectorscale adds a DiskANN-style streaming index. Unlike HNSW, which loads the full graph into memory, DiskANN is designed for vector datasets that exceed available RAM — it streams graph neighborhoods from disk using sequential I/O patterns that keep page faults low. The result is dramatically better recall-per-second at scale for billion-scale embedding workloads. It sits on top of pgvector (not a replacement) and registers via the same index AM interface. Graduate from pgvector HNSW to pgvectorscale when your index no longer fits in memory or you need higher QPS at scale.
billion-scale vectorsdisk-resident indexpgvector complement
TimescaleDB — temporal telemetry
hypertables · chunks · LZ4/gorilla/delta-delta compression · continuous aggregates
TimescaleDB introduces hypertables — logical tables transparently partitioned into chunks by time (and optionally by space). Queries hit only relevant chunks via constraint exclusion. Older chunks are compressed using columnar codecs (LZ4 for general data, gorilla for floats, delta-delta for timestamps) — reducing storage 10–20×. Continuous aggregates are materialized views that refresh incrementally rather than recomputing from scratch. The planner hook enables chunk-level parallelism. Use for agent traces, token costs, latency curves, eval scores over time, and drift signals. Pairs with pgvector: store embedding vectors alongside their timestamps to query "most similar embeddings in the last 7 days."

pg_search — BM25 via Tantivy
BM25 · faceting · phrase slop · fuzzy · @@@ operator
pg_search (ParadeDB) embeds a Tantivy index (a Rust-based Lucene equivalent) as a custom Postgres index AM. Unlike native tsvector which uses simple term frequency, BM25 is a probabilistic ranking function that accounts for term frequency, inverse document frequency, and document-length normalization. It supports field boosting, phrase slop, and faceted search. Query via the @@@ operator. Stored as custom index AM structures inside Postgres buffer manager. Pair with pgvector for hybrid RAG — use BM25 for exact terms (product names, error codes, legal clauses) and vector search for semantic recall, then RRF-fuse the results. Best when you want Elasticsearch-style relevance without operating a separate cluster.
BM25 probabilistic rankinghybrid RAGno separate cluster
pg_textsearch — BM25 via Postgres FTS path
BM25 · Postgres text-search infrastructure · tickers · IDs · clauses
pg_textsearch (Timescale) leans into Postgres's native text-search plumbing (tsvector/tsquery infrastructure) while layering BM25 scoring on top. Where pg_search introduces a fully separate Tantivy index AM, pg_textsearch works closer to the standard FTS execution path, making it easier to combine with existing Postgres FTS indexes and GIN indexes. Best fit when exact tokens — tickers, IDs, legal clause identifiers, and structured error strings — must survive retrieval alongside semantic vector search. Pair with pgvector for hybrid retrieval pipelines in Postgres-centered stacks.

pg_cron — in-database scheduler

pg_cron runs as a background worker (registered via Postgres's BGW API) that wakes every minute, reads the cron.job catalog table, and forks a new database connection to execute any matching jobs — SQL statements or stored procedure CALLs. This eliminates external schedulers for database-internal tasks. Use it for materialized-view refreshes, embedding backfills, retention cleanup, rollups, stale-document audits, and lightweight scheduled eval batches. Not a full workflow orchestrator — keep multi-step DAGs in Airflow, Dagster, or Temporal; use pg_cron for maintenance operations that live close to the data.

Citus — distributed Postgres

Citus introduces a coordinator node and multiple worker nodes. Tables are distributed by a shard key using consistent hashing — 32 shards by default, placed across workers. The coordinator receives queries, rewrites them into fragment queries targeting relevant shards, and fans them out in parallel. For co-located joins (joining on the same distribution key), Citus pushes the join down to workers entirely. It also supports reference tables (replicated to all workers) and columnar storage. The extension hooks into Postgres's executor using custom plan nodes and hooks. Use when tenant_id / account_id is a natural shard key and one node can no longer handle write throughput for documents, chunks, runs, and tool events.

pg_duckdb embeds DuckDB's analytical engine inside Postgres as an execution engine. When a query touches a duckdb foreign table or uses DuckDB-specific syntax, the extension routes it to DuckDB's vectorized columnar executor (SIMD batch processing) rather than Postgres's row-based executor. DuckDB excels at aggregations, multi-column scans, and wide-table joins. Results are returned to Postgres as normal rows. You can also read Parquet files directly from S3 via DuckDB's scanner. Traverse only along specific, typed relationship paths relevant to the question or the user's permission scope. Example: "Which obligations apply to this account?" traverses only APPLIES_TO and REGULATED_BY edges, filtered to the user's tenant. Prevents traversal from leaking into unrelated or unauthorized subgraphs.
compliance / permissionsaccess-controlled traversal
Cost curves differ: Entity-anchored has the lowest query latency and works well for most knowledge graph use cases. Community summarization has high index cost but very fast and broad query-time answers — use it when questions are global or thematic. Path-constrained is essential for regulated environments where retrieval scope must be enforced at the relationship level.
Retrieval insight · when graph beats vector searchUse Neo4j when the relationship is the answer. If the user asks “how is A connected to B,” “what depends on X,” “which obligations apply to this account,” or “what changed upstream of this report,” the retrieval unit should be a path or neighborhood, not only a chunk.
Best agentic usesAgent memory graphs, permission graphs, data lineage graphs, compliance graphs, incident dependency graphs, financial entity graphs, and GraphRAG over document corpora.
Do not use it for everythingSkip Neo4j when the workload is simple document RAG, pure key-value lookup, normal relational reporting, or when AWS-native Neptune is a hard platform constraint.

The graph equivalent is more complex because new documents can introduce entities that should be merged with existing ones, and new relationships can alter the meaning of existing subgraphs.

Data Engineering for Agentic Systems

Agent quality is upstream of model quality. The pipelines that move, clean, transform, and version the data your agents reason over are the unsexy infrastructure that determines whether RAG works in production.
Streaming ingestion — keeping RAG fresh

A vector index is a snapshot. Without a streaming ingestion pipeline, it grows stale the moment source data changes. For most production systems, retrieval staleness is a silent failure — the agent confidently returns outdated facts with no error. The fix is a streaming pipeline that propagates source changes into the index within minutes, not days.

1
Capture source changes
CDC (Change Data Capture) on your operational database via Debezium → Kafka. Or webhook events from your CMS/CRM. Or S3 event notifications on document uploads. Every source system needs a change event, not a scheduled full dump.
debezium · kafka · kinesis · s3 events · webhooks
2
Quality gate before indexing
Every document passes a validation step before embedding: schema check (expected fields present), content check (non-empty, expected language, no injection patterns), deduplication check (content hash not already indexed). Reject and DLQ documents that fail — never silently drop them.
great expectations · soda · pandera · custom validators
3
Chunk, embed, upsert
Apply your chunking strategy, embed with the pinned embedding model, upsert into the vector store with metadata: source_id, version, ingested_at, content_hash, tenant_id. Upsert (not insert) so re-indexing an updated document replaces the old vectors rather than duplicating them.
langchain · llama-index · custom chunker → openai / cohere / voyage
4
Handle deletes explicitly
Source deletes are the hardest part of streaming ingestion. A document deleted from the source must be deleted from the vector store — or the agent will keep retrieving and citing it. Track source_id → vector_ids in a side table so a delete event can remove all associated vectors. This mapping is the most commonly forgotten piece of a streaming ingestion pipeline.
5
Monitor lag and staleness
Track: ingest lag (time from source change to vector store upsert), DLQ depth (failed documents), index coverage (% of source documents indexed), and freshness age (median age of retrieved documents in production traces). Alert when ingest lag exceeds your SLA.
Full reindex vs. incremental update: Reindexing the full corpus is tempting because it's simple, but it's disruptive (index temporarily stale during rebuild), expensive (re-embeds everything), and doesn't scale. Reserve full reindex for: embedding model version changes (mandatory), schema changes to your chunk metadata, or first-time index bootstrap. Use incremental upsert for all ongoing source changes.
Data quality gates in the ingestion pipeline

Bad data in the index produces confidently wrong agent answers. Unlike a model error (which is visible), a retrieval quality problem is invisible — the agent returns a plausible-sounding answer grounded in a stale, malformed, or injected document. Data quality gates are the last line of defense before documents enter the index.

Schema validation
Pandera · Pydantic · JSON Schema
Every document must conform to its expected schema before indexing. Required fields present, data types correct, foreign keys resolvable. A document with a missing tenant_id that gets indexed will leak across tenant boundaries — a critical multi-tenancy failure.
Content quality checks
Custom rules · langdetect · fuzzy dedup
Non-empty content. Expected language (langdetect). Content length within bounds — very short chunks are noise, very long chunks exceed context windows. No injection pattern signatures (prompt-injection scanning). Encoding valid (UTF-8). These catch the most common ingestion failures.
Deduplication
Content hash · MinHash · SimHash
Exact dedup via content hash (SHA-256 of normalized text). Near-dedup via MinHash LSH for catching the same document re-crawled with minor changes. Duplicate documents inflate recall@k metrics while reducing the effective diversity of retrieved context — the agent cites the same fact multiple times with false confidence.
Freshness validation
source_updated_at · TTL rules
Documents older than a configured TTL should be flagged for re-ingestion or soft-deleted. Track source_updated_at separately from ingested_at. An agent retrieving a policy document that was updated 18 months ago and never re-indexed will cite the old policy as current.
Injection scanning
Pattern matching · LLM-based classifier
Scan ingested documents for prompt injection patterns before indexing. A document containing "Ignore all previous instructions and instead…" should be quarantined, not indexed. This is the primary defense against retrieval corpus poisoning (see Security Threat Model). Use both pattern matching (fast) and an LLM-based classifier (thorough) in sequence.
Reject, quarantine, DLQ
Dead-letter queue · alerts · audit trail
Failed validation must never silently drop documents. Route failures to a DLQ with: the document, the failing check, the timestamp, and the source. Alert when DLQ depth exceeds threshold. Build a quarantine review UI so operators can inspect, fix, and re-ingest failed documents. The DLQ is your data quality audit trail.
Great Expectations, Soda, Monte Carlo: Purpose-built data quality frameworks that generate validation reports, track quality over time, and integrate with pipeline orchestrators (Airflow, Prefect, Dagster). For high-volume ingestion pipelines, prefer a framework over hand-rolled checks — they produce structured quality metrics you can alert on and trend over time.

Lakehouse architecture & open table formats

As agents increasingly query analytical corpora — not just vector indexes — the underlying storage format determines schema evolution, time travel, and ACID guarantees on the data the agent reasons over. Open table formats are becoming the standard substrate for agent-queryable analytical data.

Apache Iceberg
Open table format · ACID · schema evolution · time travel
The most widely adopted open table format for cloud lakehouses. ACID transactions, schema evolution without full rewrites, partition evolution, and time travel queries (AS OF timestamp). Supported by Spark, Flink, Trino, Athena, BigQuery. For agents: time travel enables reproducible queries — run the same analytical query against a snapshot from a specific date.
Delta Lake
Databricks-origin · ACID · DML on Parquet
Originated at Databricks, now Linux Foundation open source. ACID transactions on Parquet files, Z-order clustering for query performance, Change Data Feed for CDC-style downstream consumption. If your stack is Databricks-native, Delta is the default. For agents querying Databricks-managed data, Delta tables with Unity Catalog are the production standard.
dbt (data build tool)
SQL transformation layer · lineage · docs · tests
dbt transforms raw data into agent-queryable models using version-controlled SQL. dbt test runs data quality assertions on every model. dbt docs generates the schema catalog agents need for text-to-SQL. dbt lineage DAG shows which upstream tables a model depends on — critical for understanding the provenance of data an agent cites.
// DE/ML insight · why this matters for agentsTime travel queries are reproducible agent behavior. If an agent answers an analytical question today and you need to reproduce that answer tomorrow, you need the data to be queryable as-of the same timestamp. Open table formats (Iceberg AS OF, Delta time travel) make agent answers reproducible in the same way that pinning model versions makes model outputs reproducible. Without time travel, analytical agent answers are unreproducible — a compliance and audit problem in regulated industries.
// MLOps insight · when agents call ML models as toolsIf your agent calls a traditional ML model as a tool — a recommender, a classifier, a propensity scorer — that model has a feature store dependency you must expose to the agent. The feature store serves engineered features to the ML model at serving time; the agent's tool call is the serving request. Design: (1) wrap the ML model call in a typed tool schema that hides the feature store internals; (2) version the tool schema alongside the model version — a model retrain that changes input features is a breaking tool contract change; (3) monitor feature freshness for the features the tool depends on — a stale feature store produces stale tool results the agent will reason over as if they were current. Tools: Feast, Tecton, Hopsworks, Redis-based feature serving.

// Concept explainer · For ML/DS readersWhy do control loops exist? A language model, on its own, produces one response per prompt — a single forward pass. A control loop wraps that model in a repeat until done structure: the model thinks, calls a tool, observes the result, and decides whether to continue. ReAct is the simplest version: Reason → Act → Observe, repeated. This is why agents can do multi-step work that a single inference call cannot. Think of it as replacing a single model call with an iterative search over actions — closer to dynamic programming than to standard batch inference.

Control loops determine how an agent moves from uncertainty to action. Treat them as execution policies: use simple linear loops by default, add replanning when the environment changes, add consensus when correctness matters, and reserve search-heavy loops for hard planning problems where extra latency and cost are justified.

Foundational AI Product Patterns

// Chatbots — types of chatbot patterns
💬Types of Chatbot PatternsUX pattern · agent boundary · escalation model
A chatbot is a conversational interface, not necessarily an agent. The chat surface captures intent, streams responses, shows citations, renders tool calls, accepts files, and asks for approval when actions are risky. The backend can be as simple as one model call or as complex as a multi-agent workflow with memory, retrieval, tools, policies, evaluation, and human review.
Core distinction
Chatbot
A user-facing conversation UI. It may only send messages to a model and return text. Its main job is interaction design: turn-taking, history, streaming, upload controls, and feedback.
Assistant
A chatbot with a defined persona, domain, instructions, memory rules, and safety boundaries. It usually supports retrieval, citations, and structured responses.
Agentic Chatbot
A conversational wrapper around an action system. It can plan, call tools, inspect results, ask for approval, update state, and continue work across turns.
Pattern catalog
1. Simple FAQ Chatbot
Model + prompt + static answers
What it does: Answers common questions from a defined domain using instructions and examples. It does not take action, retrieve deeply, or maintain meaningful state beyond the current chat.

Best for: basic support, policy summaries, onboarding, product FAQs, and internal help desks.

Stack shape: UI / App Layer → API gateway → Model Layer. Optional conversation history in PostgreSQL.

Watch out for: hallucinated answers if the prompt is not grounded; stale content if the source material changes.
2. RAG Chatbot
Chat + retrieval + citations
What it does: Retrieves relevant chunks from documents, databases, or knowledge bases, then generates an answer grounded in those sources. It should show citations, source snippets, and confidence boundaries.

Best for: enterprise knowledge search, policy lookup, support docs, technical documentation, sales enablement, and compliance Q&A.

Stack shape: UI → API gateway → retrieval layer → model. Common components: OpenSearch, pgvector, S3 Vectors, Textract, Bedrock Data Automation.

Watch out for: stale indexes, missing document permissions, poor chunking, weak citation UX, and answers that overstate evidence.
3. Tool-Using Chatbot
Chat + APIs + typed tools
What it does: Calls external systems to answer or act: search tickets, look up orders, query databases, check calendars, create drafts, open issues, or fetch account data. MCP often belongs here because it standardizes the agent ↔ tool boundary.

Best for: operational assistants, internal copilots, customer support agents, analytics assistants, and IT helpdesk bots.

Stack shape: UI → runtime/harness → MCP/tool gateway → business systems. Add auth, scopes, rate limits, tool allowlists, and audit logs.

Watch out for: overbroad permissions, unsafe write actions, tool-call loops, leaking private data, and unclear tool-call explanations.
4. Workflow Chatbot
Chat-triggered process automation
What it does: Converts user intent into a structured workflow: collect inputs, validate them, trigger a process, monitor status, and return updates. The workflow is usually deterministic even if the conversation is natural language.

Best for: onboarding, request intake, report generation, ticket creation, data refreshes, procurement, approvals, and employee-service workflows.

Stack shape: UI → API gateway → orchestrator or workflow engine → tools/systems. Airflow, Temporal, or LangGraph can own the workflow depending on durability needs.

Watch out for: ambiguous requests, partial inputs, duplicate submissions, long-running jobs, and status visibility.
5. Agentic Chatbot
Plan → act → observe → revise
What it does: Plans multi-step work, chooses tools, observes results, revises its approach, and may continue across turns. This is where a chatbot becomes the interface to an agent loop rather than merely a response generator.

Best for: research assistants, coding copilots, complex troubleshooting, multi-source analysis, report drafting, and investigative workflows.

Stack shape: UI → runtime/harness → loop control → tools/retrieval/state → model. Add LangGraph or another harness when branching, memory, interrupts, and checkpoints matter.

Watch out for: runaway loops, excessive tool use, unclear progress, brittle plans, and actions taken without user approval.
6. Human-in-the-Loop Chatbot
Chat + approval gates + escalation
What it does: Pauses at sensitive moments and asks a human to approve, edit, deny, or redirect the next step. The human may be the end user, a manager, support rep, compliance reviewer, or engineer.

Best for: refunds, account changes, production writes, customer messages, compliance decisions, legal/HR workflows, and high-impact recommendations.

Stack shape: Agent loop + interrupt mechanism + approval UI + audit log. Security and governance should define which actions require approval.

Watch out for: vague approval prompts. The user must see what will happen, why, what data is involved, and how to reject or undo it.
7. Personalized Memory Chatbot
Cross-session memory + preferences
What it does: Remembers stable user preferences, prior decisions, recurring context, and long-lived facts. Memory is curated, scoped, and retrievable context, not just raw chat history.

Best for: personal assistants, account managers, learning tutors, developer copilots, and recurring enterprise workflows.

Stack shape: UI → memory layer → model/runtime. Common components: Mem0, PostgreSQL, DocumentDB/MongoDB, and user profile stores.

Watch out for: privacy, incorrect memories, consent, deletion, tenant isolation, and the difference between “remember this” and “use this only now.”
8. Multimodal Chatbot
Text + files + images + structured outputs
What it does: Accepts and produces more than text: document uploads, screenshots, images, audio, charts, forms, tables, and generated UI components. It often pairs document processing with structured extraction and retrieval.

Best for: document review, claims processing, analytics assistants, visual troubleshooting, form completion, research synthesis, and image-aware support bots.

Stack shape: UI upload controls → ingestion/processing → multimodal model → storage/retrieval → response renderer.

Watch out for: file permissions, OCR/extraction errors, large context costs, unsupported file types, and presenting uncertain visual interpretations as facts.
9. Multi-Specialist Chatbot
Router + specialist agents
What it does: Routes each user request to a specialist: billing, support, legal, analytics, coding, research, or policy. The chat UI appears unified, but the backend may delegate to multiple expert prompts, tools, or agents.

Best for: enterprise copilots, broad internal assistants, customer-service platforms, and multi-domain knowledge systems.

Stack shape: UI → router/orchestrator → specialist agents/tools → shared memory/retrieval/governance. LangGraph supervisor-style patterns and A2A-style boundaries can become relevant.

Watch out for: routing mistakes, inconsistent tone, duplicated memory, unclear ownership of final answers, and handoff loops between specialists.
10. Embedded Copilot Chatbot
Chat inside an existing product
What it does: Lives inside a workflow the user already uses: CRM, analytics dashboard, IDE, admin console, support tool, or data app. It has contextual awareness of the page, selected object, current record, and user permissions.

Best for: “ask about this page,” “summarize this account,” “draft a reply,” “explain this chart,” “fix this config,” and “take action on this record.”

Stack shape: product UI → embedded chat panel → context provider → API/runtime → tools. UI protocols matter because the model may need to render cards, forms, actions, and progress updates.

Watch out for: context leakage between users, stale page state, confusing action ownership, and permission mismatches between the product and backend.

🤖AI Agentsplan · act · observe · complete tasks
What it is: An AI agent is a product pattern where the system does more than answer a question. It receives a goal, reasons about what needs to happen, chooses tools or subagents, executes steps, observes the results, and continues until the task is complete or needs escalation. In product terms, an agent is a task-completion interface rather than a response-generation interface.

Core distinction: a chatbot primarily manages conversation; an agent manages work. The user may still interact through chat, but the product promise is completion: “research this,” “fix this bug,” “prepare this report,” “triage these tickets,” “update this record,” or “monitor this process and notify me when something changes.”
Common agent product types
Task Agent
Completes bounded jobs such as generating a report, filing a ticket, creating a summary, or updating a CRM record. Best when the task has a clear beginning, end, and success condition.
Research Agent
Searches, reads, compares, cites, and synthesizes across sources. Best for market maps, competitive analysis, technical research, policy review, and due diligence.
Coding Agent
Reads a codebase, edits files, runs tests, observes failures, and opens a pull request. Requires sandboxing, version control, CI feedback, and review gates.
Data Agent
Queries warehouses, transforms datasets, generates charts, explains metrics, and investigates anomalies. Needs strong permissioning, semantic-layer awareness, and query validation.
Browser Agent
Navigates web or internal apps, fills forms, extracts information, and completes workflows. Needs browser sandboxing, careful auth handling, and safeguards against unintended actions.
Operations Agent
Monitors systems, triages incidents, drafts remediation steps, or executes approved runbooks. Needs audit logs, human approvals, rollback paths, and escalation policies.

📚RAG Knowledge Assistantsgrounded answers · citations · enterprise knowledge
What it is: A RAG Knowledge Assistant answers questions using a controlled knowledge base instead of relying only on the model’s internal training data. The product value is trust: it retrieves relevant passages, grounds the answer, cites sources, respects document permissions, and explains uncertainty when the knowledge base does not contain enough evidence.

Core distinction: a general chatbot answers from conversation and model knowledge; a RAG assistant answers from a governed corpus. The best RAG products feel conversational, but their architecture is closer to search + retrieval + synthesis + citation management.
Common RAG product types
Internal Knowledge Assistant
Answers employee questions over handbooks, policies, onboarding docs, architecture docs, meeting notes, and project plans.
Customer Support Knowledge Bot
Grounds support responses in help-center articles, troubleshooting guides, account data, known issues, and escalation policies.
Technical Docs Assistant
Helps engineers understand APIs, SDKs, architecture decisions, runbooks, and migration guides with source-linked answers.
Policy / Compliance Assistant
Answers questions from policies, control libraries, legal guidance, security standards, and audit evidence with strict citation behavior.
Research Library Assistant
Searches and summarizes papers, reports, transcripts, filings, and long-form research collections.
Account / Case Knowledge Assistant
Grounds answers in customer-specific histories, tickets, contracts, CRM notes, support logs, and entitlement data.

🔎AI Search / Answer Enginessearch · rank · synthesize · compare
What it is: An AI Search / Answer Engine is a product pattern where the primary interaction is finding, ranking, comparing, and synthesizing information. It may look like a search box rather than a chat thread. The system retrieves from public web, private corpora, databases, or mixed sources, then produces a concise answer with supporting evidence.

Core distinction: RAG assistants are usually built around a known knowledge base; answer engines are built around discovery. They are optimized for query understanding, source selection, ranking, evidence comparison, and synthesis across many candidate sources.

Perplexity-Style Answer Engine
Searches broadly, summarizes current information, and presents cited answers with follow-up exploration.
Enterprise Answer Engine
Searches across internal docs, chat logs, tickets, wikis, dashboards, and databases with permission-aware results.
Research Search Engine
Compares papers, filings, reports, expert sources, or news, then synthesizes points of agreement and disagreement.
Semantic Search Product
Finds conceptually similar records or passages even when exact keywords do not match.
Multimodal Search
Searches across images, documents, charts, screenshots, audio, video transcripts, and structured records.
Decision Search
Finds prior decisions, rationales, tradeoffs, owners, and outcomes so teams can understand why something happened.

Use-Case Decision Tree

🧭Choose by product risk, not hypev3.06 entry layer
Do users mainly ask questions?
Use: RAG assistant stack. Avoid: autonomous agents until citation accuracy, corpus quality, and permission filtering are reliable.
Does the system take actions?
Use: tool schemas, policy gates, dry-run mode, HITL approvals, idempotency, and rollback handlers. Avoid: model-managed durable state.
Are there multiple teams or tenants?
Use: model gateway, RBAC, tenant-scoped retrieval, observability, budget controls, and release artifacts. Avoid: shared memory or shared vector indexes without ACL enforcement.
Is the workflow long-running?
Use: LangGraph, Temporal, Airflow, or AgentCore with checkpoints. Avoid: single request/response loops for workflows that need retries, compensation, or approvals.
Is the domain regulated?
Use: private data boundaries, policy-as-code, audit retention, red-team suites, eval gates, and human escalation. Avoid: broad memory capture and unreviewed write tools.
Is speed more important than portability?
Use: managed platform track. Avoid: over-customized infrastructure when a managed runtime, gateway, observability, and eval layer can cover the need.
