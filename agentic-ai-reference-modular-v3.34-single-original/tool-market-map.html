🔁Loop Patterns & Iteration Control1 primitive · 34 loop families · search + consensus expanded
Loop control is the agent's execution policy. The primitive loop keeps an agent running; the loop family determines how it searches, acts, critiques, verifies, coordinates, remembers, escalates, and stops. These patterns are composable: a production system can use a DAG wrapper, a Plan–Execute–Reflect outer loop, ReAct inner tool calls, verifier gates, retrieval research, HITL interrupts, and multi-agent fan-out in the same run.

What git worktrees are
A native Git feature that lets you check out multiple branches simultaneously into separate working directories from a single repository. Each worktree is an isolated filesystem — different branch, different working files — but all share the same .git history. No container overhead, no duplication of the repo.
In an agent context
Spin up N worktrees → run N Claude Code instances simultaneously, each in its own isolated working directory. Parallel autonomous agent runs without branch collision. The lightweight alternative to Docker containers per agent run — near-zero overhead, just filesystem isolation via Git. Useful for parallel autonomous agent coding tasks.

✂️Chunking & Reranking — quality levers between ingestion and retrieval
Chunking strategies
How you split documents before embedding dramatically affects RAG quality — a chunk boundary in the middle of a key fact means neither chunk is useful on its own.

Fixed-size — split every N tokens, simple, fast, ignores structure
Recursive — split by paragraph → sentence → word, respects natural boundaries
Semantic — split where meaning shifts, requires an embedding model
Document-aware — extract structure first, then chunk by heading / table / section

Recommendation: document-aware for complex PDFs (section-level chunks), recursive for plain text.
Reranking
Vector search retrieves top-K candidates quickly but imprecisely. A reranker (cross-encoder) then re-scores them against the query with much higher accuracy — expensively but only over K candidates, not the full index.

Cohere Rerank — managed API, best out-of-box quality
BGE Reranker — open-source, self-hostable, strong multilingual
Cross-encoder reranking — the general pattern: retrieve 20, rerank to top 5, generate

The standard pattern in production RAG: vector search (recall) → rerank (precision) → generate.
GraphRAG chunking — different rules than standard RAG
Preserve entity boundaries
Standard fixed-size chunking splits on token count regardless of semantic boundaries. For GraphRAG, a chunk that splits mid-sentence loses the relation triple connecting two entities — the extraction step will miss or corrupt the relationship. Use sentence-boundary chunking for any corpus that will feed a graph extraction pipeline.
Keep co-referring entities together
Entity resolution works within a chunk. If "the company" appears several sentences after "Acme Corp," and a chunk boundary falls between them, the extractor loses the coreference. Use longer overlapping chunks (e.g. 512 tokens with 128 overlap) for extraction, even if you use smaller chunks for the parallel vector retrieval path.
Run two parallel chunking strategies
Optimal chunking for graph extraction and optimal chunking for vector retrieval are different. Use larger, boundary-aware chunks for extraction → load into graph. Use smaller semantic or sentence chunks for embedding → load into vector index. The same source document feeds both pipelines from a single ingestion run.
Attach source provenance to every chunk
Every graph node and relationship should carry a reference back to the source chunk it was extracted from. This enables GraphRAG to return not just the graph path but the exact text evidence behind each relationship — essential for explainability and for debugging extraction errors.
RAG & GraphRAG — patterns, not tools. RAG wires the retrieval layer into the agent: query → embed → search vector DB → retrieve chunks → generate. GraphRAG (Microsoft) extends this with graph traversal: builds a knowledge graph from documents, then combines graph hops + vector search for multi-hop reasoning. More expensive to build, dramatically better on relational questions. Embeddings are the foundation both depend on — text converted to vectors via a model (OpenAI text-embedding-3, Cohere, or local HuggingFace models).

🛡️Security, Governance & Trust3 projects · 5 concerns
Not one thing — five distinct concerns that fire in sequence. Conflating them is the most common mistake in production agent systems.
① Entitlements
What a given user or role is allowed to ask the agent to do. The claim layer — "User A can run reports but not modify schema." Fires first, before any agent logic runs.
② RBAC
Who can invoke which agents, skills, slash commands, or MCP tools. Maps roles to capabilities — analyst role can call query agent but not write agent. Implemented at harness level via tool permission gating.
③ Policies
Behavioural rules governing what the agent is allowed to do — distinct from guardrails (content) and RBAC (access). "May only read, not write." "Must request human approval before deleting." Capability restrictions on the agent itself.
④ Guardrails
Content safety at the input/output boundary. Validates what goes in and what comes out — blocks prompt injection, PII leakage, harmful outputs, off-topic responses. Fires at hook points (PreToolUse · PostToolUse · Stop).
⑤ Filtering
Data-level restriction on what the agent can see and return — distinct from all above: this is about data scope, not behavioural scope. Row-level security on DuckDB, index-level access control on OpenSearch, response redaction of fields the user isn't entitled to see.

🔌 APIs & CLIs → MCP  ·  API / CLI vs MCP — not three things in competition, two categories with a clear boundary
The correct framing: API / CLI vs MCP
APIs and CLIs are the same category — raw interfaces. A CLI is an API you call via stdin/stdout. A REST API is a CLI you call via HTTP. Both require the caller to know exactly what to call and how. Neither has a schema an agent can introspect at runtime.

MCP is a different category — a typed, discoverable protocol layer built on top of APIs and CLIs. MCP wraps them; it doesn't replace them. The underlying service still has its API or CLI. MCP is the agent-facing interface on top.

// Graph construction pipeline — from documents to knowledge graph
Building a knowledge graph from documents is a separate pipeline from standard RAG ingestion. Each step has distinct failure modes. The most common place teams get stuck with GraphRAG is not the graph database — it is this pipeline.
Six-stage construction pipeline
1 · Extract entities
Run NER over chunked text. Extract people, orgs, products, concepts, events, systems, risks. Use a fine-tuned NER model or LLM-based extraction with a typed schema.
Failure: hallucinated or over-extracted entities
2 · Extract relations
Identify typed relationships between entities: WORKS_AT, DEPENDS_ON, REPORTS_TO, OWNED_BY, CITED_IN. Each relation needs a source, target, type, and provenance span.
Failure: wrong direction, missing type, noise relations
3 · Resolve identities
Merge "MSFT," "Microsoft Corp," and "Microsoft" into one canonical node. Coreference resolution across documents. This step determines graph quality — poor resolution fragments retrieval.
Failure: duplicate nodes fragment multi-hop paths
4 · Load graph
Batch-upsert nodes and relationships into Neo4j or Neptune. Use merge semantics (MERGE in Cypher) not blind insert to avoid duplicates. Attach provenance: source document, chunk ID, confidence, timestamp.
Failure: duplicate edges, missing provenance
5 · Build vector index
Embed entity descriptions and attach vector indexes to graph nodes (Neo4j vector index or Neptune Analytics). This enables entity-anchored retrieval: embed the query → find nearest entity nodes → traverse.
Note: same model pin rules as standard RAG
6 · Incremental updates
New documents trigger re-extraction for changed entities only. Merge new relations, re-run disambiguation for affected entity clusters. Community summaries need recompute if community membership changes.
Failure: stale communities, orphaned nodes
Pipeline tooling
Entity + relation extraction
spaCy · GLiNER · LLM-based extraction (Claude, GPT) with typed output schemas · Instructor library for structured LLM output · Microsoft GraphRAG library (open source)
Entity disambiguation
String matching · embedding similarity · knowledge base linking (Wikidata, internal catalogs) · LLM-based coreference resolution · entity linkers (REL, BLINK)
Graph loading + pipelines
Neo4j Python driver (MERGE statements) · neo4j-graphrag Python library · LangChain LLMGraphTransformer · LlamaIndex property graph stores · Airflow/Prefect for orchestration
Pipeline insight · extraction quality determines retrieval qualityA knowledge graph is only as good as its extraction pipeline. The two highest-leverage investments are entity resolution (step 3) and schema design (done before step 1). A graph with fragmented duplicate nodes produces broken multi-hop paths. A graph with an ill-defined schema produces edges that the query planner cannot traverse reliably. Fix both before scaling ingestion volume.
