🚀Production Deployment Playbookarchitecture · operating model · safety · CI/CD · SLOs · rollout · rollback
Purpose: This playbook turns the agent stack into an operational deployment model. Use it to move from prototype to production with explicit architecture, release gates, monitoring, safety controls, rollout discipline, and rollback paths.

Core principle: Treat prompts, tools, models, retrieval indexes, policies, and eval datasets as production artifacts. Version them, test them, deploy them gradually, observe them continuously, and keep a known-good rollback path.

Incident response and rollback plan
Incident triggers
Unsafe output, data leakage, unauthorized tool action, retrieval permission failure, cost spike, latency breach, outage, high hallucination rate, or repeated user escalations.
Immediate containment

Context & Prompt Engineering

The highest-leverage layer. The model layer serves the context; the context layer determines everything else. Most production failures originate here.
System prompt anatomy
1 · Identity & role
Who the agent is, what it is responsible for, and what it must never do. Written once and version-controlled. Should be stable across releases.
2 · Task instructions
Exactly what the model should do with a user request. Includes output format (JSON schema, XML tags, markdown), step-by-step reasoning instructions, and negative examples.
3 · Tool manifest
Which tools are available, when to use each, and what valid invocation looks like. Describe edge cases and ambiguous overlap explicitly.
4 · Dynamic context slot
Retrieved documents, user profile, session state, and runtime data injected here. Always labeled. Never mixed with role text.
5 · Few-shot examples
Concrete input-to-output pairs showing desired behavior. Most effective for structured output, edge cases, and tone calibration. Curate from real production traces.
6 · Guardrail rules
Topics to avoid, escalation triggers, refusal language. Phrased as positive obligations ("always cite sources"). Audit when safety metrics worsen.
// ML/DS insight · in-context learningFew-shot examples exploit in-context learning (ICL): gradient-free meta-learning that temporarily biases the model's output distribution at inference time. Unlike fine-tuning (which updates weights), ICL requires no training — the model generalises from examples in its context window alone. Quality matters more than quantity: 3 carefully chosen, diverse, correctly-labelled examples consistently outperform 20 noisy ones. Curate examples from production traces where the model's behaviour was ideal, not from synthetic or hand-written cases — real query distributions produce better generalisation. ICL is also fragile to example order; if results vary, average over a few orderings or place the most representative example last.

// ML/DS insight · context as feature budgetThe context window is the agentic equivalent of a real-time feature budget in online ML. In ML serving, you can't compute expensive features in the critical path — you must prioritise the most predictive ones within a latency budget. The context window imposes the same constraint: you can't inject all available information, so you must select. The "dynamic context slot" for retrieved documents is your real-time feature store serving features at inference time. The retrieval strategy determines which features (chunks) are selected — exactly the feature selection problem in online ML. Techniques that transfer directly: entropy-based relevance scoring for chunk selection, compression of redundant features (duplicate retrieval results), and lazy loading (retrieve only when confidence is below threshold).
Prompt versioning & drift management
Store prompts in git
Treat system prompts as code. Every change goes through PR review with an eval run attached. Tag releases with semantic versions. Never edit prompts in a UI without a git-backed audit trail.
Pin prompts per environment
Dev, staging, and prod each reference a specific prompt version. Promotion is gated on passing evals. A model version bump may require a prompt rewrite even without changing the wording.
Track behavioral drift
Run the same golden-set eval weekly even if nothing changed. Models on the same version string can still drift subtly. Catch regressions before users do.
Structured output enforcement
Require JSON mode or constrained generation for any downstream-parsed output. Define the schema in the prompt with a concrete example. Validate every response against the schema at the harness layer.

Section TL;DR
  • Evals are the test suite for AI behavior - build them before you need them, gate every deploy on them.
  • Cover: correct answers, adversarial prompts, tool-use scenarios, retrieval-grounded tasks, tenant-boundary tests.
  • LLM-as-judge for open-ended output; exact match for structured output; trajectory eval for tool-using agents.

Evaluation EngineeringStable

Eval checklist
  • Separate model-quality tests from task-success tests, retrieval tests, tool-call tests, and regression suites.
  • Keep gold examples, adversarial cases, and recent production failures in the eval set.
  • Block release when safety, grounding, latency, or cost thresholds regress.
Evals are the test suite for AI behavior. Without them, you are flying blind through every model update, prompt change, and retrieval tweak. Build evals before you need them.
Anatomy of an eval case
Input
The exact user message and conversation history. Use realistic phrasing, not idealized queries.
Context
The retrieved documents or tool results the model should access. Separates retrieval failures from model failures.
Expected output
The ideal response or rubric. Can be a string match, a list of required claims, a schema, or LLM-as-judge criteria.
Metadata
Intent category, difficulty, source (production trace / synthetic / adversarial), creation date. Enables sliced eval reporting.

// ML/DS insight · embedding architectureWhy query and index embeddings must come from the same model: Embedding models are trained to map text into a specific learned vector space — a bi-encoder architecture where each piece of text is embedded independently. Similarity search only works if query and document vectors live in the same space, trained on the same objective. Switching embedding models without reindexing is the same mistake as fitting a StandardScaler on training data and applying the saved scaler parameters to data with a different feature distribution — the coordinate systems no longer align, and cosine similarity between them is meaningless. Always track the embedding model version alongside the index version, and treat any model change as a mandatory full reindex.
Embedding model selection and version management
Pin the embedding model version
The embedding model used to index must be the same used to query. Changing the embedding model requires a full reindex. Track the embedding model version alongside the index version in metadata.
Embedding drift
When you upgrade the embedding model, the existing index is stale — old vectors and new query vectors are in different semantic spaces. Build a parallel reindexing pipeline before cutover, not a blocking migration.
Dimensions and normalization
Different embedding models output different vector dimensions. Normalize vectors before indexing when using cosine similarity. Verify normalization is consistent between index and query paths — a mismatch silently degrades recall.
Cost tracking
Embedding API calls accumulate fast at ingestion. Track tokens embedded per run, cost per corpus, and incremental vs. full reindex cost. Batch embedding calls — do not embed one chunk at a time.

// ML/DS insight · reranker architectureRerankers are cross-encoders — a different architecture from the bi-encoders used for retrieval. A bi-encoder embeds query and document independently, enabling fast approximate nearest-neighbor search across your entire corpus. A cross-encoder attends jointly to both the query and document in a single forward pass — far more accurate, but computationally expensive and not scalable to full-corpus search. The standard pipeline (and the same pattern used in neural NLI and sentence-pair classification) is: bi-encoder for approximate top-k retrieval → cross-encoder to rerank only those top-k candidates. If your reranker lift (the metric below) is near zero, the first-stage retrieval quality is the bottleneck, not the reranker.

// SWE insight · backpressure & circuit breakersWhat happens when the model API returns 429, the vector store is saturated, or a tool call hangs? Model Version Management

Model updates are the most common source of silent production regressions. The model changed; your prompts, evals, and expectations did not.
// ML/DS insight · model version = distribution shiftA new model version is a new data-generating process with a different output distribution — and your existing golden dataset is a sample from the old process. This is the same problem as train/test distribution mismatch: your test set may not cover the new model's specific failure modes, especially edge cases that the old model handled correctly but the new model handles differently. Before promoting a new version, audit whether your golden dataset still has good coverage of the new model's behavioral space. New failure modes often cluster in areas that weren't failure modes before — corners of the input distribution the old model happened to handle well. If your eval suite passes but production quality drops after a model upgrade, distribution shift in the golden set is the first hypothesis to investigate.

Treat provider-forced upgrades as a production incident risk and treat A/B testing across model versions as a first-class operational capability.
// MLOps insight · shadow mode for agent upgradesShadow mode is the safest way to validate a new agent version before any user-facing cutover. Route live traffic to both current and candidate agent versions simultaneously. Serve only the current version's response to users; collect both responses for offline evaluation. Compare: answer quality (LLM-as-judge), tool call patterns, latency, cost, error rate. Promote only when shadow eval passes your quality threshold — zero users were exposed to the experiment. This is identical to shadow mode in ML model serving: a traffic splitter at the API gateway, a parallel execution path, and a comparison eval pipeline. Unlike A/B testing, shadow mode has zero user risk. Essential before any model version upgrade or major prompt change in a production system.

Caching strategy
Prompt caching
Cache the static prefix of the system prompt at the provider level (Anthropic supports prompt caching with cache_control). Reduces input token cost significantly for long system prompts reused across requests.
Semantic cache
Cache full LLM responses keyed by embedding similarity to the query. GPTCache or custom Redis + vector similarity. Cache hit rate depends heavily on query distribution — measure before assuming it helps.

// ML/DS insight · cache threshold is a precision-recall hyperparameterThe similarity threshold that triggers a semantic cache hit is a precision-recall tradeoff you need to tune, not set arbitrarily. A lower threshold returns more cache hits (higher recall) but risks returning a cached answer to a query that is similar-but-not-identical — a precision problem. Measure cache precision (% of hits where the cached answer is correct for the new query) alongside the cache hit rate. A cache with a 40% hit rate and 60% precision is actively degrading answer quality. Treat the threshold as a hyperparameter: tune it on a held-out query set, plot hit rate vs. precision as you vary it, and choose the operating point that meets your quality floor. Re-tune after significant changes to your user query distribution.
Model routing by cost tier
Not every request needs the frontier model. Route by complexity: simple classification and routing decisions → cheapest fast model (Haiku, GPT-4o-mini). Standard Q&A, summarization, RAG responses → mid-tier model (Sonnet). Complex multi-step reasoning, orchestrator decisions, long-context synthesis → frontier model (Opus, GPT-4o). Implement routing as a classifier model or rules-based system. Measure quality impact of routing decisions in evals — routing that degrades quality is not saving money, it is losing users.
// ML/DS insight · routing as a learned classifierRule-based routing is fast to build and brittle to maintain — consider training a routing classifier. As query complexity grows, a rule set becomes a decision tree that degrades under distribution shift. The scalable alternative is a lightweight intent classifier: a small fast model predicts the complexity tier and routes the request before the main model call. Build the training set from production traces labelled with route outcome and downstream quality. Deploy the classifier at the gateway — it adds single-digit milliseconds but makes routing accuracy measurable, improvable, and drift-detectable over time. Track routing accuracy the same way you track any model in production: periodic eval against labelled samples, alerting on distribution shift, and retraining when accuracy drops.
Cost governance
Hard cost caps
Enforce per-user, per-session, and per-day token budgets at the gateway. When a budget is exhausted, return a graceful "limit reached" response. Reset budgets on schedule.
Cost-aware testing
Add a cost-per-task assertion to your eval suite. If a test case that should cost $0.02 now costs $0.18, that is a regression even if quality metrics pass. Track cost per task over time in CI.
Tenant cost quotas
Track cost per tenant_id. Enforce quotas matching the tenant's pricing tier. Alert the tenant before they hit their quota. Alert your team when a tenant's cost suddenly spikes.
Cost modeling before scaling
Before scaling traffic by 10x, model the cost impact. Build a cost estimator from token usage distribution and request volume. Run this as part of capacity planning, not as a post-launch surprise.

Async Job Queue ArchitectureCurrent

Any agent run longer than ~30 seconds needs a job queue under it. Blocking HTTP connections for long-running agents is the most common production architecture mistake for agentic systems.
Why synchronous HTTP fails for agents

A synchronous agent endpoint — POST /run → wait → response — works fine for runs under ~10 seconds. Beyond that: load balancers time out, mobile clients disconnect, server restarts lose in-flight runs, and a single stuck tool call blocks the entire response. The production pattern is always async: accept the job, return a job ID immediately, run the agent in a worker, poll or push the result.

Result delivery patterns: (1) Polling — client polls GET /run/{job_id}/status until done; simple but chatty. Dead-letter queue is not optional. Every job queue needs a DLQ. Failed agent runs must land somewhere observable — not silently disappear. Configure: max receive count (retries before DLQ), DLQ depth alarm (alert when DLQ > 0), and a replay mechanism (re-enqueue DLQ jobs after fixing root cause). A DLQ is your agent system’s catch-all error handler and audit trail for failures.
Agent state checkpointing & resume

Checkpointing persists agent state after every step so a crashed or paused agent can resume from the last checkpoint rather than restarting from scratch. Essential for HITL approval gates (pause mid-run, wait for human, resume), cost control (restart only failed steps), and long multi-step runs (survive worker restarts).

1
Define checkpoint schema
Minimum fields: run_id, step_index, graph_state (serialized), tool_results_so_far, status (running / awaiting_human / done / failed), created_at. Store in Postgres or Redis with TTL.
2
Write after every tool call
Before executing the next step, persist current state. If the worker crashes between steps, the next worker picks up from the last persisted checkpoint — the write-ahead log pattern from databases applied to agent execution.
3
HITL pause/resume via checkpoint status
When the agent reaches an approval gate: write checkpoint with status: awaiting_human, surface the proposed action to the reviewer, block until status changes to approved or rejected. Worker resumes by loading checkpoint and continuing from step_index + 1.
4
LangGraph checkpointer or Temporal
LangGraph’s built-in checkpointer (Postgres or Redis backend) handles steps 1–3 automatically. Temporal’s workflow engine handles durable execution at the infrastructure level — crashes are transparent to application code. Use Temporal for multi-agent coordination across services; LangGraph checkpointer for single-agent pause/resume.
